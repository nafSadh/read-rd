How LLMs Remember

The KV Cache

How transformers trade memory for speed during inference — and why it matters for every token you generate

Scroll to explore
01 — The Attention Bottleneck

Why Every Token Matters

In autoregressive generation, each new token must attend to all previous tokens in the sequence. This is the fundamental constraint of self-attention: when you generate "mat" in the sequence "The cat sat on the mat", that final token must compute its relationship to every preceding token.

Self-Attention

Each query Q multiplies with all keys K, scaled by dimension, then softmax weights applied to values V. Formula: Attention(Q,K,V) = softmax(Q·K^T/√d)·V

Autoregressive Constraint

During inference, tokens generate left-to-right. Each new token's query must attend to all previous tokens—but cannot look ahead. This creates a causal mask and forces recomputation.

02 — Cache the Keys and Values

The Core Insight

Here's the key realization: K and V matrices for past tokens never change. Only the new token's Q, K, V need computation. We can cache all previous K and V values, reuse them for every subsequent token, and skip the recomputation entirely.

Prefill Phase

Process the prompt in parallel (compute-bound). Compute Q, K, V for all tokens at once. High throughput, compute is the bottleneck.

Decode Phase

Generate one token at a time (memory-bound). Compute new Q only, reuse cached K, V. Memory bandwidth becomes the bottleneck.

03 — Anatomy of the Cache

Memory Layout & Dimensions

The KV cache is a 4D tensor with shape [batch, num_heads, seq_len, head_dim]. As the sequence grows, memory consumption grows linearly with sequence length—but compounds across layers.

Batch Dimension

Number of sequences processed in parallel. Each sequence in a batch maintains its own independent KV cache. Multiplies total memory requirement.

Number of Heads

Multi-head attention. Each head maintains separate K and V projections. LLaMA-2 70B: 64 heads. More heads = more cache per token.

Sequence Length (Growing)

The dimension that grows dynamically. Starts small (1 token), grows as you generate output. This is the primary memory concern in long-context generation.

Head Dimension

Fixed dimensionality of each attention head. LLaMA-2 70B: 128 dimensions per head. Standard formula: model_dim / num_heads.

04 — The Memory Wall

When Cache Becomes Massive

As context length grows, KV cache memory consumption can exceed the model's weights. A 70B parameter model with 128K context can consume hundreds of gigabytes of cache—making long-context inference infeasible without optimization.

Linear Memory Growth

KV cache grows O(n) with sequence length. At 128K tokens, cache alone can be 200GB+. Becomes the primary bottleneck in long-context inference.

Model Weights Fixed

Model parameters stay constant. LLaMA-2 70B weighs ~140GB. But cache exceeds this at moderate context lengths, limiting practical deployment.

05 — Shrinking the Cache

Optimization Techniques

Multiple strategies reduce KV cache footprint without sacrificing quality. From architectural changes (MQA, GQA) to quantization and smart eviction policies.

Multi-Query Attention (MQA)

Single K and V head shared across all Q heads. Reduces cache to 1/num_heads size. Trade-off: quality degradation. Used in Falcon, PaLM.

Grouped Query Attention (GQA)

Compromise: g groups of heads share K/V. Typically 4-8x compression. Maintains quality better than MQA. Used in LLaMA-2, Gemini.

Quantization

Store K and V in INT8 or INT4 instead of FP32. 75-87% memory savings with minimal quality loss. Supported in vLLM and TensorRT-LLM.

Sliding Window Attention

Only cache recent tokens (e.g., last 4096 tokens). Effective for local patterns. Mistral 7B: 4K window. Breaks long-range dependency capture.

Token Eviction (H₂O)

Keep high-importance tokens, evict low-importance ones. H₂O paper: identify via attention scores. Maintains context quality with 50-70% cache reduction.

KV Merging

Merge semantically similar tokens' K/V. Reduces cache while preserving key information. Advanced technique, trade-off between compression and fidelity.

06 — PagedAttention: Virtual Memory for AI

The vLLM Breakthrough

PagedAttention (Kwon et al., 2023) is the killer optimization behind vLLM. It treats KV cache like OS virtual memory: logical blocks map to physical memory through a block table, enabling near-zero fragmentation and memory sharing across requests.

Block Tables

Map logical block IDs to physical memory addresses. Each request maintains its own block table. Enables non-contiguous allocation without software overhead.

Memory Sharing

Copy-on-write semantics. Requests sharing a prefix (e.g., system prompt) share the same physical blocks. When one modifies, allocate new block. Save 40-60% on common prefixes.

Continuous Batching

Preempt and resume requests without losing state. New requests start immediately; old requests pause. Eliminates idle GPU time. Vastly improves utilization.

Near-Zero Waste

Fragmentation reduced to <4% (vs 60% traditional). Because blocks are small (16 tokens) and flexibly allocated, padding/waste is minimal.

07 — The Production Landscape

How Major Frameworks Handle KV Cache

Leading inference frameworks implement KV cache optimization differently. Understanding their strategies helps you choose the right tool for your workload.

vLLM

PagedAttention, continuous batching, memory sharing. Best-in-class throughput. 10-40x speed vs naive inference. Industry standard for long-context and batch serving.

TensorRT-LLM (NVIDIA)

In-flight batching, FP8 KV cache quantization, GPU-optimized kernels. Lower latency for single requests. Native NVIDIA hardware optimization.

HuggingFace Text Generation Inference (TGI)

Flash Attention, token streaming, dynamic batching. Strong on ease-of-use. Integrates HF models seamlessly. Growing performance parity with vLLM.

SGLang

RadixAttention (prefix tree for KV), speculative decoding, multi-modal support. Research-focused. Excellent for advanced batching patterns and constrained generation.

LM Studio (Local)

Simplified KV cache management, user-friendly UI. Single-request focus. Good for development and testing. Sacrifices some throughput optimization.

llama.cpp

Memory-efficient, CPU/GPU hybrid. Basic KV caching. Focus on accessibility, not maximum throughput. Excellent for edge deployment and small models.

Future Directions

Disaggregated KV Stores: Separate KV cache from inference engine. Allows scaling caching independently from compute.

Cross-Request Caching: Share KV cache across users/requests. Radical efficiency gains for multi-user scenarios (but privacy considerations).

Speculative Decoding with Cache Prefilling: Speculatively generate tokens and precompute their KV cache. Accept when tokens match, reject otherwise. Unifies prefill/decode phases.
