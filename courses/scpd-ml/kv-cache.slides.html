§1 1

Autoregressive Generation

How LLMs generate text token-by-token through attention
I
§1 · Generation 2

Autoregressive Decoding

Generating one token at a time

LLMs don't generate entire sequences at once. They produce text one token at a time, where each token depends on all previous tokens.

Start with [BOS] → model outputs token "The" → feed back "The" → model outputs "cat" → feed back "The cat" → continue…

This is called autoregressive generation (auto = self, regressive = depends on past).

Autoregressive Formula P(token_t | token_0...token_t-1) = softmax(logits_t)
Why autoregressive?

At inference time, we can't know future tokens. So we predict one at a time, feeding each back into the model.

§1 · Attention 3

Attention Mechanism Recap

Query, Key, Value matrices

At each position, a token's output is a weighted sum of all token values.

Query (Q): "what am I looking for?" — derived from current token

Key (K): "here's what I am" — one per token in context

Value (V): "here's my information" — one per token in context

Attention Output Output = softmax(Q @ K^T / sqrt(d)) @ V
Core insight

Attention weights (softmax scores) choose which values to combine. High score = that token's info contributes more.

§1 · Complexity 4

The Quadratic Bottleneck

Why attention is O(n²) in sequence length

Attention computes scores for every pair of tokens in context: Q @ K^T is [n, d] @ [d, n] → [n, n].

For a sequence of length n, that's n² comparisons — quadratic in context length.

With 80 layers × 64 heads in LLaMA 2 70B, at 2048 tokens: 2048² × 80 × 64 = 21 billion operations per token.

Attention Cost Time: O(n²·d) Memory: O(n²) for attention matrix
The problem

Each new token requires recomputing Q@K^T and attention over the ENTIRE growing context. Huge redundant work.

§1 · Waste 5

The Redundancy Problem

Each token regenerates all previous K,V pairs

When generating token 10, you compute attention over tokens 1-10, including K and V for tokens 1-9.

When generating token 11, you compute attention over tokens 1-11, and must recompute K and V for tokens 1-9 exactly the same way.

Massive waste: 90% of computation in step 11 repeats from step 10.

As context grows, redundancy balloons: at step n, (n-1)/n of the work is replicated.

The insight

K and V are deterministic once computed. Why recompute them? Cache them!

§2 6

The KV Cache Idea

Saving computed keys and values across generation steps
II
§2 · Core Idea 7

Cache K,V Across Steps

The fundamental optimization

Instead of recomputing K,V for previous tokens, store them in memory after first computation.

At step t, retrieve cached K,V for tokens 1...t-1, compute fresh K,V for token t, concatenate, and reuse for step t+1.

Transforms attention from O(n²) computation per token to O(n) computation per token (only new token needs K,V computation).

Without cache Step n: recompute K,V for all n tokens
With cache Step n: load K,V for 1...n-1 from cache + compute for token n

§2 · Phases 9

Prefill vs Decode Phases

Two distinct stages in inference

Prefill phase: Process prompt tokens all at once (e.g., 500 tokens). Build initial KV cache. Batch dimension is high, great GPU utilization.

Decode phase: Generate output one token at a time (or small batch), append to KV cache each step. Batch is small, bound by memory bandwidth.

KV cache is built in prefill, then grown incrementally during decode.

Two-Phase Pipeline Prefill: grow cache from 0 → prompt_length Decode: grow cache from prompt_length → max_length
Why split?

Prefill is compute-bound (large batch). Decode is memory-bound (batch=1). Cache amortizes prefill cost over many decode steps.

§2 · Memory 10

Memory Layout

Shape and storage of cached tensors

KV cache is stored as: [batch, num_heads, seq_len, head_dim]

For LLaMA 2 70B: 80 layers, 64 attention heads, 128 dim per head, batch=1, max_seq=2048

Per layer cache size: 2 × 1 × 64 × 2048 × 128 bytes = 33.5 MB per layer (FP32)

Total for all 80 layers: 33.5 × 80 = 2.7 GB just for KV cache at full context!

KV Cache Memory Size = 2 × layers × batch × heads × seq_len × head_dim × precision_bytes
§2 · Speedup 11

Time Complexity Savings

From O(n²) to O(n) per token

Without KV cache: generating token t requires O(t²) compute — full attention over t tokens.

With KV cache: generating token t requires only O(t) compute — load cache, compute new K,V, one attention pass.

For 2048-token generation: without cache O(1+4+9+...+2048²) ≈ O(n³/3), with cache O(1+2+3+...+2048) = O(n²) amortized!

Speedup factor (2048²) / (2048) ≈ 2000× faster per token in 2K context
Practical impact

KV cache reduces decode time from hours to seconds on GPUs. It's the difference between usable and unusable inference.

§3 · Formula 13

KV Cache Memory Formula

Computing total memory consumption

Memory for all layers, all tokens in context:

Full Formula Cache (GB) = 2 × num_layers × num_heads × head_dim × max_seq_len × batch_size × 4 bytes / 1e9

Key insight: memory grows linearly with seq_len and batch_size, but scales with model architecture (layers, heads, head_dim).

Larger models = more cache. Longer contexts = more cache. Batch processing = more cache.

Memory vs Speed tradeoff

KV cache saves computation but costs memory. With GPU VRAM fixed, must choose batch size or context length.

§3 · Growth 15

Cache Grows Linearly

Memory consumption vs sequence length

Since cache size = O(seq_len), memory grows linearly with context.

Double context → double cache memory. Not quadratic, but still significant.

For very long documents (8K, 32K+ tokens), cache can dominate GPU memory budget.

Practical limit

On a 80GB GPU, LLaMA-2 70B can hold ~12K token context before running out of memory (FP16).

§3 · Budget 16

GPU Memory Budget

Where memory goes during inference

Model weights: ~70-280 GB for 70B-175B params (FP16)

KV cache: ~5-20 GB for 2K-8K context

Activations: ~1-5 GB per batch (intermediate layer outputs)

Total: ~76-305 GB depending on scale, context, batch.

Insight

KV cache is 5-10% of total memory but essential for speed. Trade: accept cache overhead to eliminate recomputation.

§4 17

Optimization Strategies

Reducing KV cache memory while maintaining speed
IV
§4 · Architecture 18

Multi-Query & Grouped-Query Attention

Reduce cache by sharing K,V heads

Standard MHA: Each Q head gets its own K,V pair. 64 Q heads = 64 K heads, 64 V heads.

Multi-Query Attention (MQA): All Q heads share ONE K,V pair. 64 Q heads share 1 K,V. Cache 64× smaller!

Grouped-Query (GQA): Compromise. 64 Q heads grouped into 8 groups, each group shares 1 K,V. Cache 8× smaller than MHA, 8× larger than MQA.

LLaMA 2 uses GQA with 8 KV heads for 64 Q heads. GPT-4 uses MQA.

Tradeoff

Sharing reduces cache memory but may slightly hurt model quality. Worth it for 8× speedup in decode.

§4 · Compression 19

KV Cache Quantization

Compress to INT8 or INT4

K,V values usually stored in FP32 or FP16. But don't need full precision.

FP32 → FP16: 50% memory reduction, minimal quality loss. Very safe.

FP16 → INT8: Another 50% reduction. Quantize to 8-bit integers with careful scaling. Small quality dip but huge space savings.

INT4: Extreme compression (4 bits). Requires specialized hardware/kernels. Tricky but possible.

Practical use

FP16 is standard. INT8 used when memory is critical. INT4 still experimental.

§4 · Windowing 20

Sliding Window Attention

Mistral's approach: only attend to recent tokens

Observation: LLMs mostly attend to recent context, not distant tokens.

Mistral uses sliding window attention: only cache K,V for last W tokens (e.g., W=4096).

Older tokens evicted automatically. Saves memory proportional to how much context is unused.

Reduces full-context cache but loses ability to recall very distant context. Tradeoff.

Mistral Window Cache only tokens [max(0, t - 4096), t], evict older

PagedAttention & vLLM

§5 · vLLM 22

PagedAttention Overview

Virtual memory for KV cache

Naive KV cache allocates contiguous memory. With dynamic sequence lengths, wastes up to 80% to fragmentation.

PagedAttention: Inspired by OS virtual memory. Store K,V in fixed-size blocks, not contiguous arrays.

Map logical token positions to physical blocks via block table. Non-contiguous storage looks contiguous to attention.

Result: multiple requests share prefixes without duplication. And internal fragmentation eliminated.

Key benefit

2-4× throughput improvement + dynamic batching without memory waste.

§5 · Framework 23

vLLM: Serving Engine

Production inference with PagedAttention

vLLM is an open-source serving framework built on PagedAttention.

Manages dynamic batching, KV cache efficiently, schedules requests to maximize GPU utilization.

Supports continuous batching: new requests arrive mid-batch, old ones leave. No gaps.

Achieves 2-4× higher throughput than naive batching for the same GPU memory.

Industry standard

vLLM (and similar: TensorRT-LLM, SGLang) are what power commercial LLM APIs today.

§6 · Models 25

Cache Strategies by Model

How different models balance speed vs memory

GPT-4 (OpenAI): Multi-Query Attention (MQA). Single K,V head for all Q heads. Smallest cache, fastest decode.

LLaMA 2 / 3 (Meta): Grouped-Query Attention (GQA) with 8 KV heads. Balanced 8× cache reduction.

Mistral: GQA + sliding window (4096). Both techniques combined for max efficiency.

Gemini/Claude: Proprietary, likely similar: GQA or MQA.

§6 · Frameworks 26

Inference Frameworks

Production systems for LLM serving

vLLM: Open-source, PagedAttention, best-in-class throughput. Most popular for research and startups.

TensorRT-LLM (NVIDIA): Closed-source, GPU-optimized, enterprise support. Used by major clouds.

TGI (Hugging Face): Production-focused, multi-backend, easy to deploy.

SGLang: Structured generation, state-machine scheduling for better cache reuse.

§6 · Future 27

Future Directions

Emerging techniques and research

Speculative decoding: Generate 5-10 tokens in parallel with cheaper model, verify with main model. Uses cache more efficiently.

Disaggregated KV cache: Store cache on separate memory nodes. GPU compute separate from cache memory. Improves scaling.

Cross-request cache: Share KV cache across different users' requests for common prefixes. Radical efficiency gain.

Learned index for cache: Use ML to predict which cached tokens are needed. Adaptive eviction.

Summary 28

Key Takeaways

KV Cache Problem: Without caching, decoding regenerates K,V for all previous tokens = O(n²) waste per token.

KV Cache Solution: Store computed K,V in memory, reuse across steps. Reduces decode time from hours to seconds.

Memory Tradeoff: Cache costs memory (5-20 GB for 2K context), but eliminates recomputation. Space-time tradeoff.

Optimization Techniques: GQA/MQA reduce cache 8-64×, quantization 2-4×, sliding window for long contexts.

PagedAttention & vLLM: Modern serving engines eliminate fragmentation, enable dynamic batching, 2-4× throughput gain.

Impact: Makes LLM inference practical. Enables real-time chat, long-context understanding, cost-effective deployment.
